# Convex + Vercel Migration Plan

Stand: 2026-03-20
Repo: `~/Projects/dev/padelclaw-agent`

## Zielbild

- Frontend und öffentliche HTTP-Endpunkte laufen auf Vercel.
- Persistenz, durable Workflows und serverseitige Business-Logik laufen auf Convex.
- Kein Prisma, keine SQLite-Datei, kein Docker, kein Caddy, kein PM2, kein ngrok.
- Deployment-Flow: `git push` -> Vercel Deploy + Convex Deploy.

### Wichtige Architektur-Beobachtungen

1. Die App ist logisch schon multi-tenant gedacht, das Schema ist es aber noch nicht.
   - `Booking`, `MessageLog` und `Player` haben aktuell **kein `trainerId`**.
   - `getTrainerConfig()` lädt ohne Kontext einfach `findFirst()`.
   - ICS-Feed und Trainer-Wochenansicht lesen aktuell globale Buchungen.

2. Die Meta-Webhook-Route antwortet zwar schnell, startet die eigentliche Arbeit aber nur als "fire-and-forget" Promise im selben Process.
   - Das ist lokal ok.
   - Auf Vercel ist das nicht robust genug; der Hintergrundtask muss durable ausgelagert werden.

3. Voice-Transkription nutzt ein lokales Whisper-Binary in `lib/whatsapp-media.ts`.
   - Das funktioniert auf Vercel nicht.

4. `canvas` in `lib/calendar/generate-image.ts` ist ein nativer Dependency-Risikopunkt für Vercel.

## Zielarchitektur

### Verantwortung Vercel

- Next.js App Router UI:
  - `app/page.tsx`
  - `app/onboarding/page.tsx`
  - `app/onboarding/success/*`
- Leichte HTTP-Einstiegspunkte:
  - Meta Webhook Verification `GET`
  - Meta Webhook `POST` als schneller Ack + Queue/Enqueue
  - optional ICS/Image Proxy-Routen, falls die URL unter `padelclaw.ai` bleiben soll

### Verantwortung Convex

- Datenbank
- Schema + Indexes
- Mutations/Queries für Trainer, Bookings, Players, Logs, Memory
- Actions für:
  - LLM-Calls
  - Google Calendar Calls
  - WhatsApp Send
  - Audio-Download + Transkription
- Scheduler für durable asynchrone Verarbeitung

### Empfohlene Verarbeitungskette für WhatsApp

1. Meta sendet Webhook an Vercel Route.
2. Vercel validiert/parsed Payload und antwortet sofort `200`.
3. Danach wird nur noch ein Convex-Ingest ausgelöst.
4. Convex Mutation schreibt Inbox/Event-Record.
5. Convex Scheduler startet Action für eigentliche Verarbeitung.
6. Action:
   - optional Audio transkribieren
   - Trainer/Player-Modus bestimmen
   - Agent laufen lassen
   - Tools/Calendar ausführen
   - WhatsApp Antwort senden
   - Logs persistieren

Damit hängt die eigentliche Geschäftslogik nicht mehr an der Lebenszeit eines Vercel-Requests.

## Reihenfolge der Migration

### Phase 0 - Freeze und Backup

1. SQLite-Datenbestand sichern.
2. Aktuelle `.env.local` Variablen dokumentieren.
3. Entscheiden, ob Twilio/360dialog in der Zielarchitektur erhalten bleiben oder bewusst rausfliegen.

### Phase 1 - Zielmodell in Convex definieren

1. Convex Projekt anlegen, Region **EU West (Ireland)** wählen.
2. `convex/schema.ts` erstellen.
3. Trainer-kontextualisierte Tabellen und Indexes modellieren.
4. Den bisherigen impliziten Single-Trainer-Default explizit ersetzen.

Erwartete fachliche Entscheidung:

- Entweder echtes Multi-Tenant ab jetzt.
- Oder bewusst "Single Trainer per Deployment".

Empfehlung:

- **Echtes Multi-Tenant jetzt sauber ziehen**, sonst migriert man den Architekturfehler nur weiter.

### Phase 2 - Query/Mutation-Layer bauen

1. Convex Queries/Mutations für:
   - Trainer lesen/schreiben
   - Spieler upsert
   - Buchungen lesen/anlegen/stornieren
   - Message Logs schreiben/lesen
   - Trainer Memory lesen/upserten
2. Gemeinsame Helper für Phone-Normalisierung und Zeitfenster einführen.

### Phase 3 - Datenzugriffe im App-Code ablösen

1. `lib/db/*`
2. `lib/agent/history.ts`
3. `lib/calendar/schedule.ts`
4. `lib/agent/tools.ts`
5. `app/api/calendar/[trainerId]/route.ts`
6. `app/api/onboarding/route.ts`

Ziel:

- Kein `@prisma/client` Import mehr im Runtime-Code.

### Phase 4 - Asynchrone Agent-Pipeline auf durable Jobs umstellen

1. Webhook-Ingest von Verarbeitung trennen.
2. Convex Scheduler + Action für Nachrichtverarbeitung einführen.
3. Idempotenz definieren:
   - Meta Message ID oder eigener dedupe key
4. Logging/Status pro Run speichern.

### Phase 5 - Voice Transcription ersetzen

1. `lib/whatsapp-media.ts` neu designen.
2. Audio-Download weiter über Meta Graph API.
3. Transkription in hosted STT verlagern.

Empfehlung:

- OpenAI STT (`whisper-1` oder aktueller Transcription-Endpunkt) aus einer Convex Node Action.

Warum:

- kein lokaler Prozess
- kein nativer Binary-Deploy
- kein DevOps

## Breaking Changes / Risiken

### 1. Multi-Tenancy ist aktuell fachlich inkonsistent

Größtes Risiko der ganzen Migration.

Wenn jetzt nur "Prisma durch Convex" ersetzt wird, bleiben diese Fehler bestehen:

- falscher Trainer kann geladen werden
- globale Bookings im ICS Feed
- globale Bookings in Trainerübersichten
- Player/Profile kollidieren trainerübergreifend

Diese Korrektur gehört in die Migration, nicht "später".

### 2. Vercel Request-Lebenszeit

Das historische "10s Free Tier" darf nicht die Zielarchitektur bestimmen.

Nach aktuellem Vercel-Stand:

- Docs-Update 2026-02-18: Functions unterstützen `maxDuration` in App Router Route Handlers.
- Docs-Update 2025-06-25: Hobby mit Fluid Compute hat standardmäßig/maximal 300s.

Trotzdem:

- Webhooks dürfen nicht auf langen LLM-Läufen blockieren.
- Die richtige Lösung ist durable Offloading nach Convex, nicht nur "Timeout höher setzen".

### 3. Convex Actions sind nicht transaktional wie Mutations

Wichtig für Flows mit externen Side Effects:

- WhatsApp Send
- Google Calendar Create/Delete
- STT Calls

Deshalb braucht es:

- klare Reihenfolge
- Statusfelder
- Idempotenz
- Retry-Strategie

### 4. Onboarding Rollback-Semantik ändert sich

Aktuell:

- Trainer wird erstellt
- Welcome-Nachricht wird gesendet
- bei Send-Fehler wird Trainer wieder gelöscht

Mit Convex ist robuster:

- Trainer anlegen
- `status: pending_activation`
- Welcome-Action schedulen
- bei Erfolg `status: active`
- bei Fehler `status: activation_failed`

### 5. `canvas` / native Module auf Vercel

`canvas` ist ein technisches Risiko für reproducible Builds.

### 6. Absolute URLs

Aktuell ist im Trainer-Agent der ICS-Link hart auf ngrok verdrahtet.

Das muss auf zentrale Basis-URL umgebaut werden, sonst bleiben nach Migration tote Links zurück.

## Whisper auf Vercel lösen

### Aktueller Zustand

- `lib/whatsapp-media.ts` lädt OGG herunter
- ruft lokales Whisper-Binary auf
- liest `/tmp/*.txt`

### Zielzustand

- Meta Media Download bleibt serverseitig.
- Transkription wird in eine externe API verlagert.
- Die Verarbeitung läuft in einer Convex Node Action.

### Empfohlene Lösung

- Hosted Speech-to-Text via OpenAI API

Warum diese Variante:

- kein Binary
- kein eigener Worker
- keine GPU
- keine Docker- oder VPS-Abhängigkeit

### Fallback

- Wenn STT temporär fehlschlägt:
  - userfreundlicher Platzhaltertext
  - Fehlerstatus loggen
  - optional Retry per Convex Scheduler

## LLM-Timeout Problem auf Vercel Free

### Architekturentscheidung

Nicht den kompletten Agent-Lauf im Vercel Request ausführen.

### Empfohlenes Muster

1. Vercel Route validiert Request.
2. Response sofort zurück.
3. Danach Convex Mutation/Action triggern.
4. Der eigentliche LLM-Lauf passiert in Convex.

### Warum

- robuster gegen Request-Abbruch
- sauber für Meta/Twilio Webhooks
- besser beobachtbar
- spätere Retries möglich

### Zusätzliche Härtung

- dedupe key pro eingehender Message
- Run-Status (`queued`, `processing`, `done`, `failed`)
- Fehlerlogs pro Job

## Empfohlene technische Decisions

1. Convex wird nicht nur als DB-Ersatz benutzt, sondern auch für durable async Verarbeitung.
2. Multi-Tenant-Korrektur kommt in dieselbe Migration.
3. Voice-STT wird vollständig externalisiert.
4. Vercel behält nur leichte Request-Arbeit.
5. Native Server-Artefakte werden nach erfolgreichem Switch konsequent entfernt.

## Definition of Done

- Kein Prisma Import mehr im Runtime-Code
- Keine SQLite-Datei mehr nötig
- Kein lokales Whisper-Binary mehr nötig
- Kein ngrok mehr nötig
- Kein PM2/Docker/Caddy/VPS mehr nötig
- Onboarding, Booking, Cancel, Trainer-Übersicht, Voice und Calendar laufen produktiv über Vercel + Convex
- Meta Webhook zeigt auf stabile Prod-URL
- Datenbestand wurde erfolgreich nach Convex migriert

## Externe Referenzen

- Convex Regions: EU West (Ireland) verfügbar; Region eines bestehenden Deployments ist nicht nachträglich änderbar.
- Convex Scheduled Functions: durable Scheduling aus Mutations/Actions.
- Convex Actions/HTTP Actions: geeignet für externe API-Calls.
- Vercel Functions API: App Router Route Handlers unterstützen `maxDuration`.
- Vercel Fluid Compute: Hobby standard/max 300s laut aktueller Doku.
